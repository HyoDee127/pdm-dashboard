PROJECT CASE STUDY

엑셀로 3주 걸리던 단가 변경을
버튼 3번으로 끝낼 수 있게

퍼시스에서 일룸으로 공급하는 자재 단가가 바뀔 때마다,
엑셀에 기준정보를 만들고, 수동으로 계산하고, 인쇄해서 날인하고, 스캔해서 보내던 업무를
구매담당 현업들과 함께 분석하고, ERP 안에서 전 과정이 돌아가도록 시스템을 만들었습니다.

사용자가 하는 일은 [산출] → [품의] → [합의] 버튼 3번뿐입니다.
나머지는 시스템이 알아서 합니다. 결재 절차까지 포함해도 3일이면 끝납니다.

구매팀 담당자들과 인터뷰하며 AS-IS 업무(엑셀 기반)를 분석했습니다. 가장 어려웠던 건 자재를 어떻게 분류할 것인가였습니다. 원자재(MFC), 원자재(철판/파이프), 원부자재(레진), 부자재(수입/국내), 반제품, 도급자재, AL압출제, 비철금속 — 10가지 유형마다 단가를 산출하는 기준과 연계해야 할 사업장이 전부 달랐습니다. 이걸 B45(자재유형) · B46(단가기준) · B47(합의상태) 코드 체계로 정리하는 데만 한 달이 걸렸습니다.

PROBLEM DISCOVERY
현업에서 확인한 6가지 문제
구매담당자들과 업무를 함께 따라가 보면서 발견한 문제들입니다. 겉으로는 "단가를 바꾸는 일"이지만, 안을 들여다보니 구조적인 문제가 있었습니다.
01

엑셀 기반 관리 — 정합성 보장 불가

자재기준정보, 단가, BOM을 전부 엑셀로 관리하고 있었습니다. 담당자마다 다른 버전의 엑셀을 쓰고 있었고, 수식 오류가 생겨도 알 수 없는 상태였습니다.

02

단가 변경에 3주 소요

퍼시스 자재 단가가 바뀌면 → 일룸 상품 단가에 반영하는 데 3주가 걸렸습니다. 자재→반제품→상품으로 이어지는 연쇄 영향을 수동으로 추적해야 했기 때문입니다.

3주 소요 / 10가지 자재 유형 x 4가지 단가 기준 = 조합이 너무 많음
03

통제 부재 — 누구나 바꿀 수 있음

엑셀이니까 승인 프로세스가 없었습니다. 담당자가 단가를 바꾸면 그게 그냥 반영됐습니다.

04

변경 이력 · 근거 추적 불가

단가가 왜 변경됐는지, 어떤 자재 변동 때문인지, 누가 승인했는지 기록이 남지 않았습니다.

05

단가 합의를 업체마다 건건이 수작업

협력업체와의 단가합의서를 건건이 추출하고 → 업로드하고 → 발송해야 했습니다. 합의서 한 건에 수일이 소요되고, 서명 상태 추적도 수동이었습니다.

06

BOM 변경 시 단가 연동 단절

BOM이 바뀌어도 단가에 자동 반영되지 않았습니다. 확인하고 → 피드백 받고 → 반영하는 프로세스 자체가 시스템에 없었습니다.

FOUNDATION — CODE SYSTEM
자재 분류 체계를 처음부터 만들다
이 프로젝트에서 가장 어려웠던 부분입니다. 10가지 자재 유형마다 "어떤 기준으로 단가를 산출하고, 어느 사업장과 연계해야 하는지"를 하나씩 정의해야 했습니다. 구매담당자들과 수차례 논의하며 규칙을 정리했습니다.

APPROACH
어떻게 풀었는가
"엑셀을 시스템으로 옮기는 것"이 아니라 "단가 변경의 전체 생명주기를 설계하는 것"이 핵심이었습니다.

자재 → 반제품 → 상품 자동 추적

HED(마스터)/DTL(상세)/RSN(근거) 3계층 데이터 모델로 설계했습니다. BOM을 따라 재귀 탐색하면서, 하나의 자재 단가 변동이 어떤 상품에 얼마만큼 영향을 미치는지 자동으로 계산합니다. 이 구조는 구매팀과 3번 재설계한 결과물입니다.

자재 하나를 제외 처리하면, 관련된 반제품과 상품의 단가가 자동 재계산됩니다.

반영 전에 먼저 시뮬레이션

실제 단가를 바꾸기 전에 시뮬레이션을 먼저 돌립니다. 영향도를 미리 확인하고, 제외할 품목을 선택하고, 오류를 잡은 후에야 결재를 올릴 수 있는 구조입니다.

"바꿨는데 문제가 있네" 대신 "바꾸면 이렇게 되는데, 진행할까?"

결재 + 잠금 + 이력 = 3중 통제

EP 전자결재로 승인 없이 변경 불가. 결재 상신 시 관련 상품 잠금. 모든 변경에 근거 기록. 엑셀 시절의 "아무나 바꿀 수 있는" 문제를 3중으로 차단했습니다.

변경할 수 없게 막는 게 아니라, 변경하려면 근거와 승인이 필요하게.

합의서 일괄발송으로 수작업 제거

기존에 업체마다 건건이 추출→업로드→발송하던 것을, 조건을 선택하고 일괄발송 버튼 하나로 처리하게 바꿨습니다. ClipReport로 서버에서 직접 PDF를 생성하고, DocuSign으로 양사 전자서명까지 한 번에.

합의서 10건을 보내려면 기존엔 30단계, 이제는 2단계.
ERP MODULE
유상사급 단가관리 시스템
ERP 안에 유상사급 단가의 전체 생명주기를 관리하는 모듈을 구축했습니다. 기준정보 설정 · 단가 산출 · 시뮬레이션 · 결재 · 전자계약 · BOM 연동이 하나의 흐름으로 연결됩니다.

TMI0230 자재기준정보관리

기준 설정 — 단가 산출의 출발점

자재 유형 · 단가 기준 · 연계사업장을 정의합니다. 유형을 선택하면 허용되는 사업장과 기준이 자동 필터링되고, 자재계정과 단가기준의 정합성을 실시간으로 검증합니다. 유상사급 전용 BOM(TT_PAIDBOM)도 이 모듈에서 관리합니다.

OPO0120 단가관리

컨트롤 타워 — 전체 프로세스의 허브

시뮬레이션 생성 · 자재/반제품/상품 3탭 검증 · 품목 제외 처리(변경하면 자동 재계산) · 전자결재 상신 · DocuSign 발송 · 상태 모니터링. 단가 변경의 모든 과정이 이 모듈에서 시작되고, 여기서 추적됩니다.

OPO0130 단가 시뮬레이션

사전 검증 — 반영 전에 영향도 확인

회사 · 사업장 · 자재유형을 선택하면, 프로시져가 BOM을 따라 재귀 탐색하며 대상 자재를 수집하고, 기존/변경 단가를 계산하여 3계층(자재→반제품→상품) 데이터를 자동으로 적재합니다.

OPO0140 단가합의서 일괄발송

전자계약 — DocuSign 양사 서명

기존의 "합의서 추출 → 업로드 → 업체마다 건건이 발송" 프로세스를, 조건 선택 후 일괄발송 버튼으로 한 번에 처리하게 바꿨습니다. ClipReport에서 UI를 거치지 않고 서버에서 직접 PDF를 생성하는 신규 모듈을 만들어, DocuSign twoway(양사 서명) 전자계약으로 자동 발송합니다.

TMI0220 BOM 변경관리

워크플로우 — BOM 확정 → 협력사 점검 → 피드백 → 반영

BOM이 변경되면 입고단가 의뢰를 생성하고, 사업장 담당자를 지정하여 점검을 요청합니다. 담당자가 확정입고가 · 자재비 · 가공비 · 의견을 피드백하면, 전자결재를 거쳐 최종 반영됩니다. 반영이 완료되면 상위 품목의 단가도 자동으로 재계산됩니다.

TECHNICAL CHALLENGES
풀어야 했던 문제들

자재에서 상품까지 한 번에 산출

"자재 단가가 바뀌면 반제품은? 상품은?" — BOM을 따라 재귀 탐색하면서 3가지 단가기준을 동시에 계산하는 엔진을 만들었습니다. 10단계 파이프라인을 하나의 쿼리에 녹여야 했고, 중간 결과를 단계마다 검증하며 쌓아올렸습니다.

ClipReport 서버사이드 PDF 생성

기존에는 사용자가 화면에서 리포트를 열어야 PDF가 만들어졌습니다. 일괄발송을 위해 UI를 거치지 않고 서버에서 직접 PDF를 생성하는 신규 모듈을 만들었습니다. 이 PDF를 Base64로 인코딩해서 DocuSign에 바로 전달합니다.

DocuSign OAuth + 전자서명

Access/Refresh 토큰 관리, 회사별(fursys/iloom/sidiz) 계정 매핑, twoway(양사 서명) Envelope 생성, 30일 자동 만료, 상태 폴링. 외부 API 연동은 "호출 한 번"이 아니라 "실패 시나리오를 어디까지 대비하느냐"의 문제였습니다.

10가지 자재유형 × 4가지 단가기준

유형마다 연계 가능한 사업장이 다르고, 단가기준도 다릅니다. 이 조합이 틀리면 산출 자체가 틀어지기 때문에, 유형을 선택하는 순간 허용 조합이 자동 필터링되는 검증 로직을 넣었습니다.

TRANSFORMATION
AS-IS vs TO-BE

AS-IS: 엑셀 + 수작업 + 인쇄/날인

  1. 엑셀로 자재기준정보 관리
  2. 자재→반제품→상품 영향 수동 추적
  3. 영향도 사전 분석 불가
  4. 승인 없이 담당자 임의 변경
  5. 합의서 추출 → 업로드 → 건건이 발송
  6. 서명 상태 수동 추적
  7. BOM 변경 시 단가 연동 안 됨
  8. 변경 이력 · 근거 없음

TO-BE: 시스템 자동 산출 + 전자계약

  1. 코드 체계 기반 시스템 관리 (10유형 × 4기준)
  2. BOM 재귀 탐색으로 자동 추적
  3. 시뮬레이션으로 사전 영향도 분석
  4. 전자결재 + LOCK/UNLOCK
  5. 조건 선택 → 일괄발송 (DocuSign)
  6. 상태 자동 폴링 + 서명 PDF 보관
  7. BOM 변경 → 상위 품목 자동 재계산
  8. 3계층(마스터/상세/근거) 전체 이력

LEARNINGS
이 프로젝트에서 배운 것

도메인을 모르면 코드를 짤 수 없다

유상사급 · BOM · 원가계산 · 구매/조달 프로세스를 처음 만났습니다. 구매담당자들이 쓰는 언어를 이해하고, 그걸 데이터 모델로 옮기는 과정이 개발보다 어려웠습니다. 3계층 데이터 모델은 현업과 3번 재설계한 결과물입니다.

분류 체계가 시스템의 기둥이다

B45/B46 코드 체계를 잘못 잡으면 나머지 전부가 무너집니다. 10가지 자재유형과 4가지 단가기준의 조합을 하나씩 검증하면서, "분류가 맞아야 계산이 맞고, 계산이 맞아야 시스템이 맞다"는 걸 배웠습니다.

외부 API 연동은 실패 시나리오의 싸움

DocuSign 토큰이 예고 없이 만료되고, EP 결재 상태가 비동기로 지연 도착하고, SMS API가 간헐적으로 실패합니다. "호출이 성공하면?" 보다 "실패하면 어떻게 복구하지?"를 먼저 설계해야 한다는 걸 배웠습니다.

현업과의 소통이 가장 중요했다

구매팀 담당자들이 매일 하는 업무를 같이 따라가 보지 않았다면, 엑셀을 그대로 시스템에 옮기는 데 그쳤을 겁니다. "왜 이렇게 하세요?"를 반복하면서 진짜 문제가 보였고, 그래야 제대로 된 시스템이 나왔습니다.
